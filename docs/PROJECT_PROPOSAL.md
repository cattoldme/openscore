# OpenScore 项目方案书

更新时间：2026-06-28

## 1. 项目概述

OpenScore 是一个开源、无广告、中文友好的体育赛事比分与数据查询产品。

项目第一阶段聚焦足球，提供今日比赛、赛程赛果、积分榜、球队详情、近期状态、收藏提醒和自然语言数据查询能力。

OpenScore 的目标不是做一个信息堆满屏幕的商业比分 App，而是做一个干净、可自建、可扩展、可被社区共同维护的现代化体育数据工具。

## 2. 背景与机会

现有成熟比分产品已经非常强，包括 Flashscore、SofaScore、LiveScore、365Scores，以及国内的雷速体育、捷报比分、懂球帝、直播吧等。

这些产品的问题不是功能不足，而是体验负担较重：

- 广告多
- 推送噪音大
- 博彩和盘口氛围明显
- 首页信息密度过高
- 很难按自然语言直接查询
- 不适合自建或二次开发
- 对中文用户的数据解释不够友好

开源生态里虽然有 OpenFootball、StatsBomb Open Data、OpenLigaDB、TheSportsDB 等数据资源，也有一些示例 App，但缺少一个成熟的、体验优秀的开源比分产品。

这就是 OpenScore 的切入点。

## 3. 产品定位

一句话定位：

> OpenScore 是一个开源、无广告、AI 友好的体育比分与数据查询工具。

核心关键词：

- 开源
- 无广告
- 干净
- 中文友好
- 可自建
- 数据可解释
- AI 查询
- Provider adapter 架构

## 4. 目标用户

### 4.1 普通球迷

想快速查看比赛、比分、赛程和球队状态，不想被广告和博彩内容打扰。

### 4.2 体育内容创作者

需要快速查询球队近期状态、历史战绩、积分榜、赛前信息，用于写文章、做短视频脚本或生成赛后战报。

### 4.3 开发者和开源用户

希望自建一个轻量比分工具，或在自己的网站、小程序、社区中接入体育数据展示。

### 4.4 小型赛事组织者

后续可扩展到校园赛、业余联赛、城市赛事的赛程和比分管理。

## 5. 竞品简析

### 5.1 商业比分 App

代表产品：

- Flashscore
- SofaScore
- LiveScore
- 365Scores
- 雷速体育
- 捷报比分
- 懂球帝
- 直播吧

优势：

- 数据覆盖广
- 实时性强
- 功能成熟
- 推送体系完整
- 用户规模大

劣势：

- 商业化内容重
- 广告和弹窗影响体验
- 博彩相关信息较多
- 页面复杂
- 不开源，不可自建

### 5.2 开放数据与开源项目

代表资源：

- OpenFootball / football.db
- StatsBomb Open Data
- OpenLigaDB
- TheSportsDB
- football-data.org

优势：

- 可用于 MVP 验证
- 部分数据开放或低成本
- 适合社区共建

劣势：

- 实时比分覆盖有限
- 数据完整性不稳定
- 不是完整产品
- 多数缺少中文体验

## 6. 差异化策略

OpenScore 的差异化不在于第一天就比商业产品数据更多，而在于体验和产品哲学不同。

### 6.1 无广告

默认不接广告，不做弹窗，不做信息流商业推荐。

### 6.2 无博彩导向

不做投注平台导流，不突出盘口和赔率，不把产品做成博彩辅助工具。

### 6.3 AI 查询

用户可以像问助手一样查询：

- 今晚有哪些英超比赛？
- 曼城最近 5 场状态怎么样？
- 皇马最近进球多不多？
- 哪几支队伍还在争冠？

系统返回结构化结果和简洁解释。

### 6.4 Provider Adapter

数据层不绑定单一供应商。

所有外部数据源统一抽象为 provider adapter：

- fixture provider
- result provider
- standings provider
- team provider
- player provider
- statistics provider

这样未来可以从免费 API 切换到商业 API，也可以支持社区贡献新数据源。

### 6.5 开源可自建

项目提供 Docker Compose 和清晰配置，让用户可以自建个人版或团队版。

## 7. MVP 功能范围

第一版只做足球，不做全体育项目。

### 7.1 首页

- 今日比赛
- 进行中比赛
- 即将开始比赛
- 已结束比赛
- 收藏球队的优先展示

### 7.2 比赛列表

- 联赛分组
- 开赛时间
- 主队/客队
- 当前比分
- 比赛状态
- 半场/全场结果

### 7.3 比赛详情

MVP 阶段先做轻量详情：

- 比分
- 开赛时间
- 主客队
- 比赛状态
- 基础事件
- 最近交锋，如果数据源支持

### 7.4 积分榜

- 排名
- 球队
- 场次
- 胜/平/负
- 进球/失球
- 净胜球
- 积分

### 7.5 球队详情

- 球队基础信息
- 近期 5 场赛果
- 未来赛程
- 当前联赛排名

### 7.6 收藏

- 收藏球队
- 收藏比赛
- 本地优先，后续支持账户同步

### 7.7 AI 查询

第一阶段不追求复杂预测，只做基于已有结构化数据的问答：

- 赛程查询
- 赛果查询
- 积分榜查询
- 近期状态总结
- 简洁中文解释

## 8. 非目标

第一版明确不做：

- 博彩交易
- 投注导流
- 盘口推荐
- 全体育项目覆盖
- 深度球员事件图谱
- 视频直播
- 社区论坛
- 复杂预测模型

## 9. 技术方案

### 9.1 前端

建议：

- Next.js
- TypeScript
- Tailwind CSS
- PWA
- React Query / TanStack Query

原因：

- 适合快速做移动端 Web
- SEO 和分享友好
- 后续可封装为 PWA
- 开源社区熟悉度高

### 9.2 后端

MVP 采用 Hono。

选择理由：

- 轻量、快，适合 API-first MVP
- TypeScript 类型体验好
- 路由和中间件模型清晰
- 可导出 API route type，方便前端类型化调用
- 运行时迁移空间大

NestJS 暂不采用。等业务复杂到需要更强约束的模块系统、依赖注入和企业级分层时，再评估是否迁移或局部引入。

### 9.3 数据库

- PostgreSQL 存结构化赛事数据
- Redis 做缓存和短周期比分刷新

### 9.4 实时更新

MVP 推荐 Server-Sent Events：

- 实现简单
- 浏览器支持好
- 适合比分单向推送

后续如果需要用户互动、房间、多人状态，再升级 WebSocket。

### 9.5 AI 查询

建议架构：

1. 用户输入自然语言
2. 解析意图
3. 调用内部结构化数据 API
4. 生成简洁中文回答
5. 附带表格或数据卡片

原则：

- 不让模型凭空编比分
- 所有答案必须来自本地数据库或实时 provider
- AI 只负责解释和组织语言

## 10. 数据源方案

### 10.1 MVP 候选

- football-data.org：适合欧洲主流足球赛事
- TheSportsDB：适合队伍、赛事、图片等扩展信息
- OpenLigaDB：适合部分公开联赛数据
- OpenFootball：适合历史赛程和赛果

### 10.2 数据抽象

内部统一模型：

- Sport
- Competition
- Season
- Team
- Match
- MatchEvent
- Standing
- ProviderSource
- SyncJob

### 10.3 同步策略

- 赛程：每天同步
- 今日比赛：每 5-15 分钟同步
- 进行中比赛：每 30-60 秒同步，视 API 限额调整
- 积分榜：比赛结束后同步

## 11. 合规边界

面向中国大陆用户时，必须注意：

- 不做博彩交易
- 不做投注平台跳转
- 不鼓励下注
- 不以赔率作为核心卖点
- 若展示赔率，需做严格合规评估

OpenScore MVP 建议完全不做赔率功能。

## 12. 商业化可能性

虽然项目开源，但未来可以有健康商业化：

- 托管版订阅
- 小赛事管理 SaaS
- 创作者数据工具
- 私有化部署
- 付费高级数据源配置
- 企业 Widget/API

不建议早期靠广告变现，否则会破坏项目初心。

## 14. 成功标准

MVP 成功标准：

- 用户可以在 3 秒内看到今日重点比赛
- 页面没有广告和博彩导向内容
- 至少支持一个主流足球联赛的赛程、赛果、积分榜
- 支持收藏球队
- 支持基础中文自然语言查询
- 项目可以被开发者本地跑起来

## 15. 风险与对策

### 15.1 数据实时性不足

对策：

- MVP 先降低实时性目标
- 清晰标注更新时间
- Provider adapter 方便未来替换数据源

### 15.2 API 限额

对策：

- 做缓存
- 区分冷数据和热数据
- 进行中比赛才高频刷新

### 15.3 数据版权

对策：

- 优先使用授权明确的数据源
- 不爬取商业产品私有接口
- 保留 provider attribution

### 15.4 产品范围失控

对策：

- 第一版只做足球
- 不做社区
- 不做直播
- 不做博彩
- 不追全赛事覆盖

## 16. 结论

OpenScore 有搞头，但必须避开红海打法。

正确方向不是复制商业比分 App，而是做一个清爽、开源、可自建、AI 友好的体育数据查询工具。

第一阶段目标应该是：

> 做出一个让用户愿意说“终于有个干净的比分工具了”的版本。
